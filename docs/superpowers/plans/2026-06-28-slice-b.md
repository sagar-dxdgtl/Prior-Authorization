# Slice B — production hardening + reviewer workflow + prod crypto

**Goal:** Build out the operational/compliance half of the platform on top of Slice A: real RBAC,
a human review workflow for `REVIEW` verdicts, enforced per-tenant rate limits, AWS KMS/Secrets-Manager
wiring, and a compliance-controls doc. Same patterns as Slice A (tenant-scoped RLS, `get_context`,
encrypted PHI, audited, TDD, tests green).

**Scope (this pass) → ships as small self-reviewed PRs:**
- **B1 — RBAC**: roles `admin | reviewer | user`; `require_role` dep; admin endpoints.
- **B2 — Review workflow**: `review_cases` + `review_notes` tables (RLS); auto-open a case on a
  `REVIEW` verdict; reviewer endpoints (list/claim/note/resolve).
- **B3 — Rate-limit enforcement**: DB-backed per-tenant daily/monthly quota → real `429`; headers
  reflect real remaining.
- **B4 — AWS KMS + Secrets Manager**: KMS-unwrap the Fernet keys at boot when configured; finalize
  `AwsSecrets`; mock-tested; gated (no AWS → local behavior unchanged).
- **B5 — Compliance doc**: `docs/compliance/controls.md` (SOC2 CC + HIPAA §164.308/.312 → implementation).

**Deferred to a later slice (documented, need infra/decisions):** multilingual i18n; distributed
(Redis) limiter; async batch/roster queue (SQS/Celery); patient-notification delivery (email/SMS).

**Security invariants (unchanged from Slice A):** every PHI/tenant row carries `tenant_id` + RLS; new
writes go through `tenant_session`; `tenant_id`/`role` come from the JWT, never the request body; no
plaintext PHI in new tables (link to `eligibility_checks` for the encrypted columns; store only
`member_id_hash`); no `str(exc)` leaks; tests green (`pytest -m "not live and not db"` + `-m db`).

## B1 — RBAC

- `network_probe/auth/deps.py`: add `require_role(*roles)` returning a dependency that calls
  `get_context` then `403` if `ctx.role not in roles`. (`role` already in the JWT + `users.role`.)
- Admin endpoints (new router `api/admin.py`, mounted under `/api/admin`, all `Depends(require_role("admin"))`):
  - `GET /api/admin/users` → list users in the caller's tenant (id, username, role, must_change_password,
    locked_until) — **no password hashes**.
  - `GET /api/admin/audit?limit=N` → recent `eligibility_checks` rows for the tenant, **PHI-redacted**
    (action, payer_key, npi, status, created_at, actor_id, member_id_hash — never the `*_enc` columns).
- Seed: the demo admin already has `role="admin"`. Add a reviewer demo user in the seed (role
  `reviewer`, must_change_password) for testing the workflow.
- Tests (db): `user` token → 403 on `/api/admin/*`; `admin` token → 200 + correct rows; tenant isolation
  (admin of tenant A doesn't see tenant B users).

## B2 — Review workflow

**Migration `0005_review_workflow`** (tenant-scoped + FORCE RLS + policies + grants to `preauth_app`,
exactly like the existing tables):
- `review_cases(id uuid pk, tenant_id uuid, eligibility_check_id uuid null, payer_key text, npi text,
  member_id_hash text null, status text [open|in_review|resolved] default 'open', assignee_id uuid null,
  resolution text null, created_at, updated_at)`.
- `review_notes(id uuid pk, tenant_id uuid, case_id uuid fk→review_cases, author_id uuid, text text,
  created_at)`.
- RLS policy `tenant_id = current_setting('app.tenant_id', true)::uuid` (FORCE), grants to `preauth_app`.

**Auto-open:** in `domain/audit.write_audit` (or the eligibility path), when
`result.network_status == REVIEW` and `action == "eligibility"|"network"`, insert a `review_cases` row
(status `open`, `member_id_hash` only — link via `eligibility_check_id`). Idempotent-ish (one case per
check row). No plaintext PHI.

**Endpoints** (`api/review.py`, `Depends(require_role("reviewer","admin"))`, all `tenant_session`):
- `GET /api/review/cases?status=open` → list cases for the tenant.
- `POST /api/review/cases/{id}/claim` → set `assignee_id=ctx.actor_id`, `status=in_review`.
- `POST /api/review/cases/{id}/notes` `{text}` → append a `review_notes` row (author = actor).
- `POST /api/review/cases/{id}/resolve` `{resolution, status?}` → `status=resolved`, store resolution;
  audit the action.
- `GET /api/review/cases/{id}` → case + its notes.

Repos in `db/repo.py` (`ReviewCaseRepo`, `ReviewNoteRepo`) — `tenant_id` from context only.

Tests (db): an eligibility check that yields REVIEW opens a case; reviewer claims → in_review; add
note → appears; resolve → resolved; a `user` (non-reviewer) → 403; tenant isolation.

## B3 — Rate-limit enforcement

**Migration `0006_usage_counters`:** `usage_counters(id uuid pk, tenant_id uuid, period_type text
[day|month], period_key text, count int default 0, unique(tenant_id, period_type, period_key))` +
FORCE RLS + grants. (Counter rows hold no PHI; still tenant-scoped for isolation.)

`api/quota.py`: `enforce_quota(ctx)` dependency used by metered routes (`/api/eligibility`, `/api/check`,
`/api/check-from-report`):
- Atomic upsert-increment day + month counters for the tenant (`INSERT … ON CONFLICT (…) DO UPDATE SET
  count = usage_counters.count + 1 RETURNING count`), inside `tenant_session`.
- Limits from settings (`QUOTA_DAILY`, `QUOTA_MONTHLY`, defaults 1000/20000) — per-tenant override
  possible later. If the post-increment count exceeds the limit → `429 {"message":"quota exceeded"}`.
- Update `RateLimitHeadersMiddleware` (or a response hook) so `x-quota-*` headers reflect the **real**
  DB counts for the tenant (replace the in-memory counters; keep `x-ratelimit-*` advisory).

Tests (db): set a low limit via settings override → N requests pass, N+1 → 429; headers show real
remaining; tenant A's usage doesn't affect tenant B.

## B4 — AWS KMS + Secrets Manager

- `core/secrets_provider.py`: finalize `AwsSecrets` (exists) — already pulls `preauth/<name>` from
  Secrets Manager; add a unit test with a fake boto3 client.
- `core/crypto.py`: add `resolve_fernet_keys(raw_keys, kms_client=None)` — when `FERNET_KEYS_KMS=true`
  and AWS is configured, treat each `FERNET_KEYS` entry as a base64 **KMS-wrapped** key and `kms.decrypt`
  it to the real Fernet key; otherwise pass through. `audit.py` / config use the resolved keys.
  Mock `kms.decrypt` in tests (no real AWS).
- `config.py`: add `fernet_keys_kms: bool = False`. `get_secret` already prefers env then AWS.
- All gated: **no AWS creds → identical local behavior** (the Slice-A path). Mock-tested; never calls
  real AWS in tests.

Tests: `resolve_fernet_keys` passthrough when KMS off; with a fake KMS client, wrapped→unwrapped key
round-trips a Fernet encrypt/decrypt; `AwsSecrets.get_secret` returns the secret string from a fake
client and `None` on error.

## B5 — Compliance doc

`docs/compliance/controls.md` — a controls matrix mapping **SOC2** (CC6 logical access, CC7 system ops,
CC8 change mgmt) and **HIPAA Security Rule** (§164.308 administrative, §164.312 technical: access
control, audit controls, integrity, transmission security) to the concrete implementation in this repo
(RLS, Fernet-at-rest + KMS, OAuth2/JWT + RBAC + lockout, PHI-redacted audit log, TLS/SSRF, secrets
management), plus the **gaps/Slice-C+ items** (BAA, key rotation cadence, DR, pen-test).
